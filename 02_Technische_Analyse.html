02Architektur & State Management

Das CV verwaltet einen komplexen Zustand ohne Framework. Alle Phasen laufen über CSS-Klassen auf document.body – das ist das komplette State-System.

State Flow – body.className
Phase 1: boot-phase → Schwarzer BG, Linux-Boot-Messages scrollend
Phase 2: retro-bg os-mac → Eigenes Wallpaper geladen, OS erkannt
Phase 3: retro-active os-mac → Spotlight, Terminal-Overlay aktiv
Phase 4: mode-deckblatt → removeOverlay() · CV-Inhalt erscheint
Phase 5: mode-design / mode-ats / … → setMode() Wechsel per Toggle oder Console

Kein React. Kein Vue. Kein State-Management-Framework. Trotzdem: saubere Zustandsübergänge, keine Race Conditions, keine doppelten Trigger.

setMode() – Die globale API

setMode(mode) ist als globale window-Funktion verfügbar. Alle 7 Modi sind aus der Browser-Console steuerbar. Das ist kein Zufall – das ist eine bewusste Schnittstelle.

Interne Mechanik: Body-Klassen entfernen → neue setzen → updateToggleActive() → updateFavicon(). Eine Funktion steuert alles. Single Responsibility, richtig umgesetzt.

03OS-Erkennung & Plattform-Theming

Beim ersten Load liest navigator.userAgent das Betriebssystem und setzt sofort eine Body-Klasse. Danach ist das gesamte CSS kaskadierend – kein JavaScript steuert mehr die Optik, das übernimmt der Browser.

OS Detection → Theme → Content → Sprache
Win → os-win → Windows.jpg · winget install · Taskbar · =- Progressbar · "Kaffee ist fertig."
Mac → os-mac → Mac.jpg · brew install · Dock · █░ Progressbar · "Coffee is ready."
Linux → os-linux → brauner Hintergrund · kein Dock/Menü · echo · apt install · Strg+Alt+T Konsole · bash · Ubuntu-Theme
iPhone → os-iphone → 2.5_iphone/ · Boot-Skip · DFU · Linux-Kernel-Kompakt · Mac-Desktop-Crash · .ipsw
Android→ os-android → 2.4_android/ · Boot-Skip · Homescreen+Kacheln · Notification → Chat „Jürgen Mann“ · Log.d auf Deckblatt · APK-Installer · echte Uhr/Akku

Die Plattform-Anpassungen sind tief: nicht nur Theme, sondern auch Sprache (Coffee vs. Kaffee), Progressbar-Zeichen, Terminal-Wrapper (brew install / winget install / apt install), Deckblatt-Ausgabe (Mac/iPhone print(", Win/Linux echo ", Android Log.d("CV", ") und Fensterdekoration (macOS Aqua vs. Ubuntu Dots).

Wer auf Mac öffnet, sieht ein Mac-Fenster. Wer auf Windows öffnet, sieht cmd.exe. Das Gehirn registriert nicht den Unterschied – es registriert: das fühlt sich echt an.

Warum das das primäre Easter Egg ist

Ein Linux-Kernel bootet auf einem iPhone im DFU-Modus, während im Hintergrund ein Mac-Desktop sichtbar ist, betrachtet in einem Mac-Browser. Drei Betriebssysteme gleichzeitig. Nur wer Linux kennt, erkennt die Absurdität – und denkt: dieser Mensch weiß was er tut.

Das ist kein Gag. Das ist ein Easter Egg: es sieht echt aus, bis man genau hinschaut.

05Installer-Phase & Coffee-Maker-Architektur

Die Terminal-Simulation nutzt async/await über Promises. Jede Zeile, jeder Fortschrittsbalken, jeder Rollback ist ein eigenes await. Das Timing ist choreografiert, nicht zufällig.

animateProgressBar + animateRollback
✓ ui/ux-module 0% → 15%
✓ xcode-toolchain 15% → 30%
✓ firebase-sdk 30% → 45%
✓ android-sdk 45% → 60%
⟳ coffee-maker-integration 60% → 72% → ERROR → animateRollback(72, 60)
⟳ coffee-maker-integration 60% → 72% → ERROR → animateRollback(72, 60)
⟳ coffee-maker-integration 60% → 72% → ERROR → animateRollback(72, 60)
✓ coffee-maker [HTML enabled] 60% → 100%

animateRollback(72, 60) ist das exakte Pattern aus echten Transaktionssystemen. Error → Detect → Rollback → Retry. Nicht nur ein Gag – ein demonstriertes Architekturprinzip.

06Typewriter-Engine & Typo-Mechanik

Die Typewriter-Funktion unterstützt absichtliche Tippfehler mit automatischer Zeichenweiser-Korrektur. Die Parameter werden als data--Attribute im HTML gesetzt.

typeWriter() mit Typo + backspaceChars()
data-typo-after="Produkte " ← Punkt wo Typo beginnt
data-typo-wrong="machen beide verbinden" ← falsche Version

1. Tippt: "Produkte " → typo.after erreicht
2. Tippt: "machen beide verbinden" → typo.wrong
3. backspaceChars(N, 55ms/Zeichen) → zeichenweise löschen
4. Tippt: "müssen beides verbinden." → korrekte Version

Timing: 55ms pro Backspace. 36ms Typing-Speed. 480ms Pausen nach Satzenden. Dieselbe Error → Rollback → Retry-Philosophie wie der Kaffee-Maker – aber im Deckblatt-Typewriter.

07Dynamisches Favicon-System

Das Favicon ändert sich mit jedem View-Modus. Kein externer Request. Das SVG wird zur Laufzeit als String gebaut, per btoa(unescape(encodeURIComponent(svg))) in Base64 enkodiert, und direkt als Data-URI gesetzt.

getFaviconUri(mode) → 8 verschiedene SVGs
deckblatt → JM design → HO onepage → S1
ats → ATS profilplus→ PR anschreiben→ AN
komplett → KO drucken → 🖨️ (echter SVG-Pfad, kein Text)

Der Browser-Tab lebt mit. Niemand bemerkt es bewusst – aber der Tab ist nie statisch. Beim Druckdialog: echtes SVG-Drucker-Icon, gezeichnet als Pfade zur Laufzeit.

08DevTools-Erkennung – Dual-Methode

Das Console Easter Egg triggert nur wenn DevTools wirklich offen sind. Zwei unabhängige Erkennungsmethoden, IIFE-gekapselt, einmaliger Trigger per fired-Flag.

isDevtoolsLikelyOpen() – zwei Methoden
// Methode 1: Fenstergröße (angedockte DevTools)
Math.abs(window.outerWidth - window.innerWidth) > 160
Math.abs(window.outerHeight - window.innerHeight) > 160

// Methode 2: debugger-Timing (undocked / floating)
t0 = performance.now(); debugger; dt = performance.now() - t0;
return dt > 120; ← debugger wurde pausiert = DevTools aktiv
Was die Console zeigt

⚠️ ERROR 404: BUGS NOT FOUND – roter Hintergrund, 18px, styled mit %c.

console.groupCollapsed("SYSTEM CHECK") – aufklappbar. Inhalt: STATE → aktuell stabil · CONTEXT → browser runtime · SHIP → releasefähig.

Abschluss: direkte Ansprache + Hint: try setMode('ats'). Wer DevTools öffnet um Fehler zu suchen, findet: einen Entwickler der seine Besucher erwartet hat.

09Kiosk-Mode & Session-State

Das CV versucht beim ersten User-Gesture (Klick oder Keydown, { once: true }) in den Fullscreen-Modus zu wechseln. Der Installer kündigt es an: "Kiosk mode aktiv – Klick oder Taste = Vollbild."

tryFullscreen() + markKioskUsed()
addEventListener("click", () => tryFullscreen(true), { once: true });
addEventListener("keydown", () => tryFullscreen(true), { once: true });

→ sessionStorage.setItem("kioskUsed", "1")
→ body.classList.add("kiosk-used")
→ .deckblatt-kiosk-hint: display none, dauerhaft

sessionStorage merkt sich den Fullscreen-Hinweis für die Session. Einmal weggedrückt, erscheint er nicht mehr – auch nach Reload. Session-State ohne Backend, ohne Cookie.

10iPhone-Zweig – 2.5_iphone/index.html

Der iPhone-Zweig ist eine eigenständige Narrative-Ebene. Während Mac/Win/Linux einen Installer durchlaufen, erlebt das iPhone einen DFU-Recovery-Flow – komplett mit echten Apple-Dateiendungen, iOS-internen Paketen und einem inszenierten Systemcrash.

iPhone Flow – 7 Phasen
1. iphone-boot-intro → Linux-Kernel (12 Zeilen), endet: Speicherfehler
2. iphone-systemfehler-phase→ Mac-Desktop erscheint (#1a1a1a BG, gecrasht)
3. systemfehler-overlay → "DFU konnte nicht gestartet werden" / "Systemfehler" / "DFU erneut starten"
4. iphone-dfu-screen → SVG Kabel+iPhone, "DFU gestartet. Wiederherstellung beginnen."
5. iphone-install-screen → MobileInstallation / SpringBoard / UIKit / .ipsw + coffee-maker.ipa
6. iphone-dfu-story → Story-Text im DFU-Look
7. iphone-flow-prompt → Flow-Modus, Touch-Buttons statt Keyboard
iOS-spezifische technische Details

ideviceinstaller -i jürgen-mann.ipa – echtes Open-Source-CLI-Tool. .ipsw – echtes Apple System-Recovery-Format. .ipa für den Kaffee-Maker bewusst anders als die Skill-Pakete. Pakete sind iOS-Systemframeworks: MobileInstallation (interner Installer-Daemon), SpringBoard (Home-Screen-Prozess), KeychainServices, UIKit.

Die iphone-systemfehler-phase-CSS-Klasse schaltet #mac-menu-bar, #desktop-icons und #os-dock-bar auf display: flex !important – der Mac-Desktop bricht durch den schwarzen Screen, Hintergrund #1a1a1a statt Wallpaper. System gecrasht, Finder läuft noch.

Vier Betriebssysteme gleichzeitig: Linux (gecrasht) · Mac-Finder (noch aktiv) · iPhone (DFU-Modus) · Mac-Browser (betrachtet das Ganze). Das ist kein Designentscheidung – das ist eine technische Absurdität die trotzdem funktioniert.
11UX-Entscheidungen – Anti-Dark-Patterns
Deckblatt: Navigation erst danach

Bottom-Nav und Hamburger erscheinen erst nach Ende des Typewriters (deckblatt-typing-done). Auf allen mobilen Views (Mac, Win, Linux, Android, iPhone) – Deckblatt bleibt ungestört.

Flow Mode [n]

"No problem – manchmal passt es nicht." Kein confirm(). Kein Retry. ♪-Button bleibt still. Entscheidung respektiert, Tür bleibt offen.

Musik: 30% Lautstärke

Audio startet bei 30%, nicht 100%. Der Nutzer wird nicht erschreckt. Ergänzen, nicht überlagern.

Watermarks: aria-hidden

Alle .watermark-code-Elemente tragen aria-hidden="true". Screenreader überspringen Dekorcode. Barrierefreiheit auch im Hintergrund.

ATS-Mode: sauber

Keine Easter Eggs, keine Animationen, kein JavaScript im ATS-View. Reiner Text, semantisches HTML. ATS-Parser bekommen exakt was sie brauchen.

Wallpaper-Preload

Bild vorgeladen mit Timeout. Bei Fehler: Fallback auf Retro-Türkis (#008B8B). Kein broken image, kein FOUC.

Print-Dialog

Eigener Dialog: Wahl zwischen ATS und Design. Favicon wechselt auf Drucker-Icon. Nach Close: zurück zum vorherigen Modus.

12Ehrliche Gap-Analyse
Was technisch fehlt

TypeScript statt vanilla JS. CSS-Methodologie (BEM, Modules). Unit Tests. Automatisierter Build. Bundle-Optimierung. Keine i18n-Strategie.

Warum das kein Problem ist

Das sind Entscheidungen, keine Defizite. Alles in einer Datei, kein Framework, zero-dependency: offline-fähig, maximal portabel, kein Build-Schritt nötig.

Wer diese Datei auf Framework-Standards bewertet hat den Kontext nicht verstanden. Die Architekturentscheidungen erfüllen die Anforderungen zu 100%.
13Einschätzung für Tech-Leads
  • State Management verstanden: Kein Framework, trotzdem saubere Zustandsübergänge ohne Race Conditions.
  • Async/await beherrscht: Die gesamte Installer-Phase ist eine choreografierte Promise-Kette ohne Callback-Hell.
  • CSS-Kaskade gezielt eingesetzt: OS-Theming über Body-Klassen, kein JavaScript für visuelle Änderungen wo CSS reicht.
  • Performance bedacht: Wallpaper-Preload mit Fallback, Favicon als Base64 ohne externen Request, { once: true } für Event-Listener.
  • Nutzer respektiert: aria-hidden auf Dekor-Elementen, ATS-Mode sauber, kein Dark-Pattern beim Flow-Mode.
  • Systemdenken: Error → Rollback → Retry in Kaffee-Maker, Typo-Korrektur und Rollback-Mechanik. Dieselbe Philosophie, drei verschiedene Kontexte.
Das Fazit

Das ist kein Tutorial-Code. Kein Copy-Paste-Projekt. Tausende Zeilen pro Plattform die zeigen wie jemand denkt der versteht: Nutzerverhalten ist vorhersagbar, State ist verwaltbar, Humor und Architektur schließen sich nicht aus.

Wer das gebaut hat kann in einem echten Team Code schreiben dem andere vertrauen.
